## 🔴 问题汇总

### 问题 1：Page() 定义缺失
**发现时机：** 第一次编译  
**根因：** batch-upload.js 只有方法定义，缺少小程序必需的 Page() 包裹和 data 初始化  
**修复：** 补全 Page() 结构、data 字段、生命周期函数  
**工时损失：** ~10 分钟

**🎯 预防措施：**
```markdown
【小程序页面开发检查清单】
□ Page() 包裹
□ data 初始化（所有 WXML 绑定的字段）
□ onLoad 生命周期函数
□ 所有 WXML bindtap 绑定的方法
□ usingComponents 声明（如有组件）
```

---

### 问题 2：事件处理方法缺失
**发现时机：** 点击"选择图片"按钮  
**根因：** WXML 绑定了 9 个事件（chooseImages、deleteImage、previewImages 等），但 JS 只实现了 2 个  
**修复：** 补全所有 9 个方法  
**工时损失：** ~10 分钟

**🎯 预防措施：**
```bash
# 开发时自动检查 WXML 绑定的方法
grep -o 'bindtap="[a-zA-Z]*"' pages/batch-upload/batch-upload.wxml | sort -u
# 对比 JS 中是否存在对应方法
```

**工具脚本建议：**
```javascript
// scripts/check-bindings.js
// 自动检查 WXML 绑定与 JS 方法是否匹配
```

---

### 问题 3：变量名错误（result vs resultList）× 3 处
**发现时机：** 代码审查  
**根因：** 
1. `editRecord` 方法：`const item = result[index]` 应为 `resultList[index]`
2. `onEditConfirm` 方法：`if (result[index])` 应为 `resultList[index]`
3. `saveAll` 方法：`result.filter` 应为 `resultList.filter`

**修复：** 全局搜索替换  
**工时损失：** ~15 分钟

**🎯 预防措施：**
```markdown
【变量命名规范】
□ 使用有意义的变量名（resultList 而非 result）
□ 保持命名一致性（同一文件中不混用）
□ 提交前全局搜索关键变量名
```

**工具建议：**
```bash
# 提交前检查
git diff --cached | grep -E "^\+.*result[^List]"  # 检查是否误用 result
```

---

### 问题 4：数据结构不一致
**发现时机：** 代码审查  
**根因：** ocrBatch 返回的数据未标准化，saveRecord/saveAll 中重复转换  
**修复：** 在 ocrBatch 中统一标准化数据结构  
**工时损失：** ~15 分钟

**🎯 预防措施：**
```markdown
【数据流设计原则】
□ 云函数返回数据结构明确定义
□ 前端只接收、不转换（或在入口处统一转换）
□ 编写数据结构文档/TypeScript 类型定义
```

**建议的数据结构文档：**
```javascript
// OCR 识别结果
{
  index: number,
  status: 'success' | 'error',
  icon: string,
  statusText: string,
  data: {
    productName: string,
    price: number,
    quantity: number,
    categoryId: number,
    platform: string,
    orderTime: string
  } | null,
  error: string | null
}
```

### 问题 5：stats.js 月度趋势计算 bug
**发现时机：** 代码审查  
**根因：** 每次循环重新计算最大值，导致高度计算错误  
**修复：** 分两步：先收集数据，再统一计算  
**工时损失：** ~10 分钟

**🎯 预防措施：**
```markdown
【算法审查要点】
□ 循环内是否有重复计算
□ 是否可提取到循环外
□ 边界条件（空数组、除零）
```

---

### 问题 6：云函数缺少数据验证
**发现时机：** 代码审查  
**根因：** record 云函数直接写入数据库，未验证必填字段和范围  
**修复：** 添加完整验证逻辑  
**工时损失：** ~10 分钟

**🎯 预防措施：**
```markdown
【云函数开发检查清单】
□ 必填字段验证
□ 数据类型验证
□ 取值范围验证
□ 枚举值验证
□ 权限校验（_openid）
```

## 📝 经验教训

### 1. "慢就是快"
花 10 分钟写设计文档，节省 30 分钟修复 bug。

### 2. 自动化优于人工检查
变量名错误、方法缺失等可通过工具自动发现。

### 3. 审查前置
与其事后审查，不如开发前定义清楚：
- 数据结构
- 接口规范
- 边界条件

### 4. 单一职责
batch-upload.js 承担了太多职责（UI、OCR、保存），应该拆分。
