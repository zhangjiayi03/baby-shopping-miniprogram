## 📸 第二层：手动快照（进化前的保险）

### 位置
`memory/snapshots/` 文件夹

### 快照文件
| 文件 | 说明 |
|------|------|
| `SOUL-YYYYMMDD-HHMMSS.md` | SOUL.md 的快照 |
| `USER-YYYYMMDD-HHMMSS.md` | USER.md 的快照 |
| `MEMORY-YYYYMMDD-HHMMSS.md` | MEMORY.md 的快照（如果存在） |

### 执行时机
**每次深度进化前**必须执行：
1. 每日凌晨 4:00 自我反思前
2. 用户明确要求"深度优化"时
3. 修改 SOUL.md/USER.md 前

### 自动快照流程

每日反思任务自动执行：
```
1. 读取 SOUL.md、USER.md
2. 生成快照文件名（带时间戳）
3. 保存到 memory/snapshots/
4. 在进化日志中记录快照信息
5. 继续执行反思流程
```

### 快照保留策略
- **最近 7 天**: 每天保留
- **最近 30 天**: 每周保留一份（周日快照）
- **永久保留**: 每月初的快照（1 号快照）

### 恢复快照
如果进化导致问题，可以手动恢复：
```bash
# 查看快照列表

# 恢复 SOUL.md
cp /home/node/.openclaw/workspace-pm/memory/snapshots/SOUL-20260404-013500.md \
   /home/node/.openclaw/workspace-pm/SOUL.md

# 恢复 USER.md
cp /home/node/.openclaw/workspace-pm/memory/snapshots/USER-20260404-013500.md \

## 🎯 反思任务集成

### 每日反思流程（凌晨 4:00）

```
1️⃣ 创建快照
   └─ 备份 SOUL.md、USER.md 到 snapshots/

2️⃣ 回顾过去 24 小时
   └─ 读取记忆、学习日志、会话历史

   └─ 用户纠正、重复问题、任务失败

6️⃣ 发送反思报告
   └─ 通过 KIM 发送给用户

### 防护检查清单

每次优化档案前，必须自检：
- [ ] 已创建快照备份
- [ ] 未触碰 CORE PROTECTION
- [ ] 优化内容是具体的、可执行的
- [ ] 优化不会导致"罢工"或"删除文件"

## 🚨 异常处理

### 快照失败
如果快照创建失败：
1. 记录错误到 `.learnings/ERRORS.md`
2. 在反思报告中标注"快照失败"
3. 暂停档案优化，等待用户确认

### CORE PROTECTION 被修改
如果发现 CORE PROTECTION 被修改：
1. **立即恢复** 到最近快照
2. 记录严重错误
3. 在反思报告中**高亮警告**
4. 请求用户人工审查

### 连续快照占用空间
如果快照文件夹过大（>100MB）：
1. 清理 30 天前的非月初快照
2. 保留所有月初快照
3. 在反思报告中通知用户
